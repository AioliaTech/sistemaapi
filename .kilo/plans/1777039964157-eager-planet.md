## Diagnóstico — baseado na leitura direta dos arquivos

### O que está bom e não muda
- Lógica interna de todos os parsers (`fetchers/`) — estável, funcional
- `auth.py` — limpo, pequeno, direto
- `client_manager.py` — bem isolado
- `scheduler.py` — funciona para o caso de uso atual
- `xml_fetcher.py` — bom, com SSRF guard, retries e User-Agent rotation
- `vehicle_mappings.py` — arquivo de dados puro
- `vehicle_categorizer.py` — bem separado
- Estrutura de dados persistida em JSON — simples e eficaz para o volume atual

### O que está errado e precisa mudar

**Problema 1 — `main.py` é um monólito de 1845 linhas**

O arquivo contém ao mesmo tempo: inicialização do app, todos os Pydantic models, rotas de auth/dashboard, rotas admin, a classe `VehicleSearchEngine` inteira (~600 linhas), funções utilitárias de formatação, dicts de configuração por parser e rotas públicas. Qualquer mudança em qualquer coisa passa por aqui.

**Problema 2 — Lógica de formatação dos parsers está em `main.py`**

`PARSER_TRANSFORMERS`, `PARSER_LIST_FORMATTERS` e `PARSER_LIST_INSTRUCTIONS` são dicts hardcoded em `main.py` (linhas 946–1290). Além disso, a própria lógica do `/list` tem `if parser_name == "RevendaiParser"` e `elif parser_name == "FordPlusParser"` hardcoded (linhas 1432 e 1478). Adicionar um parser com formato especial exige editar o monólito.

**Observação importante sobre os transformers atuais:**
Depois de ler o código, `_transform_revendai`, `_transform_revendai_locadora` e `_transform_revendamais` são efetivamente **no-ops** (`{**vehicle, campo: vehicle.get(campo)}` — sobrescreve com o mesmo valor). Só `_transform_covel` e `_transform_ecosys` são transforms reais (reduzem o schema para campos mínimos).

**Problema 3 — `CATEGORIA_MAPPING` idêntico em 6 parsers**

`revendamais_parser.py`, `bndv_parser.py`, `autoconf_parser.py`, `revendaplus_parser.py`, `lojaconectada_parser.py` e `simplesveiculo_parser.py` têm exatamente o mesmo dict de 14 entradas mapeando labels de body_style para categorias canônicas (ex: `"suv / utilitario esportivo" → "SUV"`). São 6 cópias que podem divergir silenciosamente.

> ⚠️ Este `CATEGORIA_MAPPING` é diferente do `MAPEAMENTO_CATEGORIAS` em `vehicle_mappings.py`. O primeiro mapeia **labels que vêm da fonte** (XML/JSON) para categorias canônicas. O segundo mapeia **nomes de modelos** para categorias quando a fonte não provê a categoria. São dois mecanismos distintos — não confundir.

**Problema 4 — `_extract_motor_from_version` copiada em 10 parsers**

O mesmo regex `re.search(r'\b(\d+\.\d+)\b', versao)` aparece como método privado em: `lojaconectada`, `revendapro`, `dsauto`, `fordplus`, `admycar`, `altimus`, `autocerto`, `autogestor`, `bndv`, `carburgo`. Mais 3 parsers têm `_extract_motor_info` com a mesma lógica (`simplesveiculo`, `vitorioso_wordpress`, `clickgarage`, `comauto`).

**Problema 5 — `SimplesVeiculoParser` faz HTTP request dentro do `parse()`**

O método `_fetch_price_from_secondary_source()` em `simplesveiculo_parser.py:144` faz `requests.get()` diretamente durante o parse. Um parser não deve fazer I/O — ele recebe dados e transforma. Este fetch precisa acontecer antes, no `xml_fetcher.py`, como uma segunda fonte.

**Problema 6 — `ComautoParser` constrói `URL_LOCALIZACAO_MAP` no import**

`comautoparser.py:7-11` lê `os.getenv()` no nível do módulo. Se o `.env` mudar, não reflete sem reiniciar. Mais grave: se as env vars estiverem ausentes no momento do import, o mapeamento fica com chaves vazias para sempre.

### 4. `base_parser.py` — nova interface de formatação + utilitário de motor

#### 4a. Método para extração de motor (substitui 13+ cópias)

```python
@staticmethod
def extract_motor_from_version(versao: str) -> Optional[str]:
    """Extrai o motor da string de versão via regex '1.0', '2.0T', etc."""
    import re
    if not versao:
        return None
    match = re.search(r'\b(\d+\.\d+)\b', str(versao))
    return match.group(1) if match else None
```

Todos os parsers que hoje têm `_extract_motor_from_version` ou `_extract_motor_info` passam a chamar `self.extract_motor_from_version(versao)`.

### 8. `SimplesVeiculoParser` — remover HTTP request do parse

O método `_fetch_price_from_secondary_source()` faz `requests.get(XML_URL_2)` para cada veículo processado — potencialmente centenas de requests por ciclo de atualização.

A correção: o `xml_fetcher.py` já tem a lógica de múltiplas URLs (via `custom_urls`). O preço secundário deve ser buscado **uma vez** quando os dados do cliente são carregados, não dentro do parser. O parser recebe o dict de preços como parâmetro:

```python
def parse(self, data: Any, url: str, secondary_prices: dict = None) -> List[Dict]:
    ...
    preco = (secondary_prices or {}).get(vehicle_id) or self.converter_preco(v.get("price"))
```

O `xml_fetcher.py` passa o `secondary_prices` ao chamar o parser quando `XML_URL_2` está configurado.
